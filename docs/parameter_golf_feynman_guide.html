Understanding Parameter Golf

This page explains parameter golf in plain language you can learn from, reason about, and teach to someone else. It combines three things in one place: a visual mental model, a glossary of key terms, and a step-by-step walkthrough of a real training-and-compression loop.

One-sentence summary

Parameter golf is the art of making a tiny language model unusually good at compressing and predicting text under strict limits on size and training time.

1. Visual mental model

The cleanest way to understand the contest is this: a language model is a text prediction machine, and good text prediction is closely related to good compression. Parameter golf asks: how much prediction power can you squeeze into a tiny, cheap-to-train package?

Mental model A

A model is a compressor

If the model can guess the next token well, it needs fewer bits to describe the text. That is why the competition score is bits per byte (BPB).

Mental model B

Parameters are memory

The model's weights are its learned memory. More weights usually help, but here storage is capped, so every weight must earn its keep.

Mental model C

Training is tuning

We are not hunting for one magical trick. We are building a loop: propose an idea, train, measure, compare, keep the good ones.

Why the Mac mini matters

Local runs are a wind tunnel

Our local MLX setup is not the final contest environment. It is a cheap screening loop. We use it to reject bad ideas fast and only promote promising ones to expensive cloud runs later.

Why quantization matters

The shipped model is the scored model

In this contest, the stored artifact matters. A model that looks great in float precision but falls apart after int8 compression can still lose.

A simple analogy

Imagine building a race car with strict rules: tiny fuel tank, strict weight limit, and a short race. You do not win by adding more engine. You win by getting every part to work together efficiently. That is parameter golf.

The three tensions that define the game

Every design choice is a tradeoff across three pressures. Good parameter golf work means learning to see these pressures clearly.

Pressure 1

Learning ability

The model needs enough capacity to learn useful text patterns. Wider layers, more layers, better tokenization, and better training all push in this direction.

Pressure 2

Storage efficiency

The final artifact must fit under the byte cap. Quantization, weight tying, and compact serialization all help here.

Pressure 3

Training speed

The model only gets a short training window. Slow ideas that need many extra steps may not matter if they cannot pay off before the wall-clock cap ends.

2. Glossary of key terms

The goal here is not textbook-perfect wording. It is to make each term intuitive enough that you can explain it to someone else without hiding behind jargon.

Artifact
The final file you submit: model weights plus any code or metadata needed to use them. The contest size cap applies to this shipped package.
Attention
The mechanism that lets a transformer decide which earlier tokens matter most when predicting the next token.
Baseline
The starting model or recipe you compare new ideas against. You need a baseline so you know whether a change helped or hurt.
Bits per byte (BPB)
A compression-style score. Lower is better. If the model predicts text well, it needs fewer bits on average to explain each byte.
Embedding
A learned table that turns discrete tokens into vectors of numbers the network can process.
Eval-time compute
Extra computation allowed during evaluation. In this contest, that can create room for clever tricks beyond ordinary training.
Gradient
A signal telling each parameter which direction would reduce the model's error.
Harness
The repeatable experiment wrapper around the model: commands, logging, summary outputs, and result collection.
Head / KV head
One attention sub-channel. Multiple heads let the model track different relationships at once. KV heads are the key/value-sharing side of grouped attention.
Layer
One repeated block of computation in the transformer. More layers mean more “thinking steps,” but also more parameters and time.
Loss
A training score the optimizer tries to minimize. For language models, it usually measures how wrong the predicted next-token probabilities were.
MLX
Apple's machine learning framework used here for local Apple Silicon experiments.
Optimizer
The algorithm that updates the model's parameters after seeing the gradients.
Parameter
A learned number in the model. Think of parameters as the model's stored memory of the patterns it has discovered.
Post-quant BPB
The score after the model has been quantized and packed into the shipped artifact. This is often more important than pre-quant quality.
Pre-quant BPB
The score before final compression. Useful, but not the full story.
Proxy run
A cheap local experiment that imitates the real evaluation loop closely enough to screen ideas quickly.
PyTorch
A popular deep learning framework. The main contest baseline trains in PyTorch on large GPUs.
Quantization
Storing weights in lower precision, such as int8 instead of float32, to save space. It can shrink the model but may hurt quality.
Quantization-aware training (QAT)
Training the model while simulating later quantization effects so it learns to survive the final compressed form better.
Quantization gap
The difference between pre-quant and post-quant quality. A smaller gap means the model survives compression better.
Recurrence / layer tying
Reusing the same weights across multiple layers or time steps so you get more effective depth without paying for all-new parameters.
Serialization
Turning the model into a saved byte representation on disk.
Token
A chunk of text the model operates on. It may be a word piece, subword, byte-level chunk, or something else depending on the tokenizer.
Tokenizer
The system that splits raw text into tokens before the model sees it.
Transformer
The core neural network architecture behind modern language models. It processes token sequences using attention and feed-forward layers.
Validation
Measuring the model on held-out data without updating its weights. This tells us how well it generalizes.
Wall-clock cap
A hard time limit measured in actual elapsed time, not number of training steps.

3. Step-by-step walkthrough of one actual run

This section walks from raw text all the way to a compressed artifact, using the same mental path we use when running experiments in this repo.

1

Start with text

The training dataset is just a large pile of text. The model does not directly understand characters or words as humans do. It only sees sequences of token IDs.

2

Tokenize the text

A tokenizer converts text into compact IDs. Example: a sentence becomes a sequence such as [412, 87, 901, 5, ...]. This is the language model's input alphabet.

3

Form training examples

The model is shown a context window, such as 1024 tokens, and asked to predict the next token at each position. That creates many prediction tasks from the same text stream.

4

Embed tokens into vectors

Token IDs are looked up in an embedding table, producing vectors. This turns discrete symbols into continuous numeric representations the network can manipulate.

5

Run the transformer layers

Each layer mixes information across tokens and refines the internal representation. Attention helps each position decide which earlier positions matter for the next prediction.

6

Predict the next token

The final hidden state is projected into logits over the vocabulary. After converting logits to probabilities, the model says how likely each next token is.

7

Compute loss

The correct next token is known from the dataset. The loss measures how surprised the model was by the truth. Lower surprise means lower loss and usually lower BPB.

8

Backpropagate and update weights

Gradients are computed, then the optimizer nudges the weights. This is how the model slowly turns from random guesses into useful predictions.

9

Validate on held-out text

We periodically stop training updates and measure how well the model predicts unseen validation data. This gives us loss and BPB without contaminating the model with the answers.

10

Quantize and serialize

After training, the weights are compressed into a smaller saved form, such as int8 with scales. This is where quality can drop if the model is not quantization-friendly.

11

Run the exact roundtrip eval

The compressed artifact is loaded back in and evaluated again. This gives the true post-quant score of the shipped model, which is what matters most for the contest.

12

Compare against the baseline

We do not keep a change because it feels smart. We keep it only if it improves the measured result enough to be worth the added complexity.

Why our current experiments focus on quantization

A model can have a strong pre-quant score and still lose points once it is compressed. That makes quantization robustness a high-leverage first target: if we can reduce the damage caused by packing the weights, we improve the score that actually ships.

What we are doing in this repo right now

We are not jumping blindly into leaderboard runs. We are building and using a local screening loop to test ideas cheaply before moving them to more expensive hardware.

Stage What it does Why it matters
Local proxy harness Runs short, repeatable MLX experiments on the Mac mini. Lets us reject bad ideas quickly and cheaply.
Baseline runs Measure the current model without modifications. Provides a control for comparison.
Quantization-focused mutations Changes training or serialization to reduce post-quant damage. Targets the final shipped score directly.
Promotion to cloud later Move the winners to the true contest setup. Local wins are promising signals, not final proof.

4. How to teach this to someone else

The Feynman method says: if you really understand something, you can explain it simply. Here is a script you can use almost word for word.

A teachable version

Parameter golf is a contest where you try to make a very small language model unusually good at predicting and compressing text. The rules limit both the final file size and the training time, so you cannot just make the model bigger. Instead, you look for smarter tradeoffs: better use of parameters, better compression, and better training efficiency. A big part of the challenge is that the final model has to be packed into a tiny artifact, which can hurt quality. So one of the smartest things to optimize is not just how good the model is before compression, but how well it survives compression.

If someone asks: “Why is BPB the score?”

Say: “Because predicting text well and compressing text well are closely related. If the model is less surprised by the text, it needs fewer bits to describe it.”

If someone asks: “Why not just use more parameters?”

Say: “The contest caps the artifact size. More parameters only help if you can fit and train them efficiently.”

If someone asks: “Why are you testing on a Mac mini?”

Say: “Because local runs are cheap and fast enough to screen ideas. We only want to spend expensive cloud time on the ideas that already look promising.”

If someone asks: “What are you optimizing first?”

Say: “The final post-quant score, because the compressed artifact is what actually gets judged.”

Common beginner confusion: “Is lower BPB really better?”

Yes. BPB is like “how many bits does the model need on average to explain the text?” Fewer bits means better predictions and therefore better compression.

Common beginner confusion: “If the local score is worse than the leaderboard score, are we failing?”

No. Local proxy runs are for direction-finding, not final ranking. They are a wind tunnel, not the race itself.

Common beginner confusion: “Why measure both pre-quant and post-quant?”

Because a model can look good before compression and then degrade after packing. The contest cares about the shipped artifact, so post-quant quality matters most.

A compact mental checklist

Ask 1

Did the model learn?

Look at training and validation quality.

Ask 2

Did it survive compression?

Compare pre-quant and post-quant BPB.

Ask 3

Did it fit the cap?

Check artifact bytes, not vibes.

Ask 4

Was it worth it?

Keep only changes that improve the shipped score enough.

One last simple summary

We are trying to build a tiny language model that stays smart even after being squeezed into a small package. To do that, we first built a fast local test loop, then we measure not just how well the model learns, but how well it survives compression. That is the heart of parameter golf.
